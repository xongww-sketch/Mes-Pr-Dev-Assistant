# 问题路由引擎

> 版本：1.0 | 最近更新：2026-04-14
> 用途：统一入口，判断问题类型，路由到对应SOP

---

## 1. 路由规则

收到任何问题，按以下规则判断类型并路由：

### 1.1 关键词匹配路由

| 关键词 | 问题类型 | 路由到 | 优先级 |
|--------|---------|--------|--------|
| bug/报错/失败/异常/出错/挂了/崩了 | Bug | bug-triage | 自动判断P0-P3 |
| 需求/新功能/要不要做/能不能加 | 需求 | requirement-analysis | P2/P3 |
| n8n/workflow/工作流/队列/worker | n8n运维 | n8n-ops | 自动判断 |
| SN/序列号/生成/绑定/解绑 | SN管理 | sn-management | 自动判断 |
| 样机/试用/领用/退款/到期 | 样机管理 | sample-machine | 自动判断 |
| 工单/work order/排程 | 工单管理 | bug-triage → 查n8n-process-map | 自动判断 |
| 渠道/OMS/同步/京东/抖音 | 数据同步 | n8n-ops → 查对应workflow | 自动判断 |
| 打印/条码/彩盒/贴纸 | 打印管理 | bug-triage → 查n8n-process-map | 自动判断 |
| 质检/FQC/OQC/不良品/返修 | 质量管理 | bug-triage → 查n8n-process-map | 自动判断 |
| 包装/出库/发货/箱号 | 包装管理 | bug-triage → 查n8n-process-map | 自动判断 |
| 设备/资源/离线/在线/登录 | 设备管理 | bug-triage → 查n8n-process-map | 自动判断 |
| 报表/看板/查询/统计 | 报表查询 | bug-triage → 查n8n-process-map | P2/P3 |
| 账号/密码/权限/token | 系统管理 | n8n-ops → 查team-context | 自动判断 |
| 数据库/慢查询/索引/PostgreSQL | 数据库 | n8n-ops | P0/P1 |
| 服务器/部署/docker/重启 | 运维 | n8n-ops | P0/P1 |

### 1.1.1 边缘情况关键词

当消息包含以下关键词但主关键词不明确时，辅助判断：

| 关键词 | 可能场景 | 辅助路由 |
|--------|----------|----------|
| 卡住/不动 | Worker挂死/队列堆积 | n8n-ops |
| 超时/慢 | 慢查询/大workflow | n8n-ops → 查get_jobmessage |
| 重复/多次 | 重复触发/幂等问题 | bug-triage |
| 丢失/找不到 | 数据未写入/SN不存在 | bug-triage → 查station_record |
| 锁了/锁定 | 分布式锁冲突 | bug-triage → 查Redis |
| 同步失败 | 外部API问题 | n8n-ops → 查SAP/OMS |
| 打印失败 | 打印机/模板问题 | bug-triage → 查打印workflow |
| 扫码失败 | 扫码枪/解析问题 | bug-triage → 查扫码workflow |
| 权限/不能操作 | 权限配置问题 | n8n-ops → 查team-context |

### 1.2 上下文判断路由

如果关键词匹配不明确，按上下文判断：

```
1. 消息来自哪个群？
   - MES·EL → 可能是Bug或运维
   - Mes开发需求对接 → 可能是需求
   - 信息化告警 → 可能是运维告警
   - WMS问题反馈 → 可能是仓储相关
   - 售后问题支持 → 可能是维修/售后

2. 消息中@了谁？
   - @熊旺 → 产品/需求相关
   - @王晓虎 → 技术/运维相关
   - @王爵 → 开发相关（SN/打印/报表）
   - @黄启立 → 开发相关（看板/质检/包装）
   - @姚骏 → 工厂/产线相关

3. 消息紧急程度？
   - 包含"紧急"/"停线"/"全部" → P0
   - 包含"帮忙看下"/"有问题" → P1/P2
   - 包含"建议"/"能不能" → P3
```

## 3. 紧急程度自动判断

### 3.1 P0 紧急（立即处理）
**触发条件**（满足任一）：
- 消息包含"停线"、"全部"、"挂了"、"崩了"、"紧急"
- 涉及n8n worker全挂
- 涉及数据库不可用
- 涉及全部产线不可用

**行动**：
1. 立即诊断
2. 通知王晓虎 + 熊旺
3. 按n8n-ops或bug-triage SOP处理

### 3.2 P1 高（30分钟内）
**触发条件**（满足任一）：
- 核心功能不可用（SN生成、绑定、过站）
- 影响部分产线
- 影响工单/出库

**行动**：
1. 30分钟内诊断
2. 通知熊旺
3. 按对应SOP处理

### 3.3 P2 中（2小时内）
**触发条件**：
- 非核心功能异常
- 有替代方案
- 报表查询慢

**行动**：
1. 2小时内处理
2. 记录到待办

### 3.4 P3 低（当天）
**触发条件**：
- 体验问题
- 优化建议
- 文档更新

**行动**：
1. 当天处理
2. 记录到待办

## 4. 多问题处理

如果一条消息包含多个问题：

```
1. 拆分问题 → 每个问题独立路由
2. 按紧急程度排序 → P0 > P1 > P2 > P3
3. 逐个处理 → 先处理最紧急的
4. 汇总输出 → 所有问题的诊断结果
```

---

## 5. 无法判断时

如果无法判断问题类型：

```
1. 回复用户：请补充以下信息
   - 问题类型（Bug/需求/咨询）
   - 影响范围
   - 复现步骤（如果是Bug）
2. 同时尝试从上下文推断
3. 如果仍无法判断 → 通知熊旺人工判断
```

## 6. 路由示例

### 示例1：Bug报告
```
消息："SN绑定失败了，产线等着呢"
→ 关键词：SN、绑定、失败、产线
→ 类型：Bug
→ 紧急程度：P1（影响产线）
→ 路由：bug-triage → sn-management
→ 输出：诊断SN绑定相关workflow + 处理建议
```

### 示例2：需求咨询
```
消息："能不能加个功能，工单关闭前自动检查物料"
→ 关键词：加功能、工单、物料
→ 类型：需求
→ 紧急程度：P3（新功能）
→ 路由：requirement-analysis
→ 输出：需求评估 + 决策框架判断
```

### 示例3：n8n运维
```
消息："n8n的worker又挂了，所有流程都卡住了"
→ 关键词：n8n、worker、挂了、所有流程
→ 类型：n8n运维
→ 紧急程度：P0（全部不可用）
→ 路由：n8n-ops
→ 输出：紧急处理流程 + 通知王晓虎
```

### 示例4：样机管理
```
消息："这个样机到期了，要不要收回？"
→ 关键词：样机、到期、收回
→ 类型：样机管理
→ 紧急程度：P2
→ 路由：sample-machine
→ 输出：样机状态查询 + 处理建议
```
