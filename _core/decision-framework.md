## 1. 核心决策原则

Jackson做决策时最常用的三条判断原则（按优先级）：

### 原则1：这个对系统有没有必要？
- **判断标准**：是否解决实际问题？是否影响核心业务流程？
- **反面案例**：为了"好看"而加的功能、没人用的报表
- **正面案例**：阻断性Bug修复、产线停线问题、数据一致性保障

### 原则2：以后好不好维护？
- **判断标准**：代码/流程是否清晰？新人能否快速理解？修改影响范围是否可控？
- **反面案例**：n8n中无文档的复杂流程、硬编码的配置
- **正面案例**：模块化设计、有文档的SOP、清晰的错误处理

### 原则3：设计符不符合长线发展和使用？
- **判断标准**：是否能支撑未来1-2年的业务增长？是否符合系统架构演进方向？
- **反面案例**：临时方案变永久、绕开MES的执行行为
- **正面案例**：数据库分层、n8n收敛定位、核心业务迁移到新MES

## 2. 5类最常见工作问题

### 2.1 SN数据缺失
**现象**：同步软件部失败，SN数据不完整
**常见原因**：
- n8n workflow异常（同步中断）
- 数据库写入失败
- 新旧MES数据不一致
**处理思路**：
1. 确认缺失范围（哪些SN、哪个时间段）
2. 检查对应n8n workflow状态（SNgenerate_API、同步类workflow）
3. 检查数据库记录
4. 判断是否需要手动补数据
**决策点**：
- 影响工单/出库？→ P0立即处理
- 仅影响报表？→ P2排期处理

### 2.2 渠道缺失
**现象**：拉取日本渠道和国内渠道失败
**常见原因**：
- OMS接口变更
- Token过期
- 渠道配置未同步
**处理思路**：
1. 检查渠道同步workflow（[Sync]系列）
2. 检查Token状态（updateLarkToken、京东Token）
3. 检查OMS接口连通性
4. 手动触发同步验证
**决策点**：
- 影响订单/发货？→ P0立即处理
- 仅影响查询？→ P2排期处理

### 2.3 MES需求问题
**现象**：需求对接、分析、优先级判断
**处理思路**：
1. 用三条决策原则评估
2. 确认需求归属（MES边界内还是外？）
3. 评估影响范围和优先级
4. 编写PRD或退回
**决策点**：
- 在MES边界内 + 有必要 + 好维护 → 接
- 不在MES边界 → 退回或转交
- 有必要但不好维护 → 重新设计

### 2.4 写需求PRD
**现象**：产品需求文档撰写
**PRD要素**：
- 业务背景和目标
- 功能概述和范围
- 流程描述（流程图 + 步骤）
- 数据设计（表结构/字段）
- 交互设计
- 异常处理
- 待确认事项
**决策点**：
- 需求是否清晰？→ 不清晰先调研
- 涉及哪些系统？→ 确认边界
- 有无历史类似需求？→ 参考已有PRD

### 2.5 维修反投售后的机型数据问题
**现象**：售后维修数据反投产线，机型数据不一致
**常见原因**：
- 工艺流程变动，旧测试工序不兼容
- 反投前测试数据未正确解绑
- 维修决策需要拆机时无法解绑
**处理思路**：
1. 确认反投SN和机型
2. 检查绑定状态（select_all_by_sn）
3. 检查测试数据状态
4. 判断是否需要解绑/重绑
**决策点**：
- 影响产线测试？→ P0处理
- 仅影响数据追溯？→ P1处理

## 3. 判断决策树

```
收到问题
  │
  ├─ 是Bug吗？
  │   ├─ 产线停线/全部不可用？ → P0 → 立即处理 + 通知王晓虎
  │   ├─ 核心功能不可用？ → P1 → 30分钟内处理 + 通知Jackson
  │   ├─ 非核心功能异常？ → P2 → 2小时内处理
  │   └─ 体验问题？ → P3 → 当天处理
  │
  ├─ 是需求吗？
  │   ├─ 在MES边界内？
  │   │   ├─ 对系统有必要？
  │   │   │   ├─ 以后好维护？
  │   │   │   │   ├─ 符合长线发展？ → ✅ 接，写PRD
  │   │   │   │   └─ 不符合 → ⚠️ 重新设计
  │   │   │   └─ 不好维护 → ⚠️ 重新设计
  │   │   └─ 没必要 → ❌ 退回
  │   └─ 不在MES边界 → ❌ 退回或转交
  │
  └─ 是咨询吗？
      ├─ 有文档/知识库？ → 直接回答
      ├─ 需要查数据？ → 查数据库/n8n流程
      └─ 需要找人？ → 查团队上下文
```

## 4. 经验教训（踩过的坑）

### 4.1 n8n相关
- **Worker全挂事件**（2026-03-21）：n8n worker全部挂掉，所有业务流程停滞
  - 教训：生产服务必须有自动重启能力
  - 教训：不要只看表象（队列堆积→实际是worker挂了）
  - 教训：应急响应需要预案（服务器账号、操作流程提前准备）

- **高频workflow堵死队列**：`resourceOnlineStatus`高频触发占满并发槽
  - 教训：高频工作流必须有熔断/限流机制
  - 教训：并发槽容量需要根据实际负载调整

- **慢SQL导致产线延时**（2026-03-19）：SQL执行时间19-20秒
  - 教训：先解决实际问题，不要过度追求重构
  - 教训：数据库性能监控必须提前建立

### 4.2 数据相关
- **新旧MES数据不一致**：部分workflow仍在使用旧MES
  - 教训：迁移必须有明确的退役计划
  - 教训：双写期间必须有数据校验机制

- **渠道同步失败**：OMS接口变更未通知
  - 教训：外部接口变更必须有通知机制
  - 教训：同步失败必须有告警

### 4.3 流程相关
- **工单关闭无审批**：曾出现随意关闭工单
  - 教训：关键操作必须有审批流程
  - 教训：权限管控不能依赖自觉

- **样机领用流程混乱**：PM领用项目问题样机无追溯
  - 教训：流程改造必须双写（PG+多维表格）
  - 教训：提高可靠性和查询性能

## 6. 决策模式总结

| 场景 | 判断依据 | 行动 |
|------|---------|------|
| 产线停线 | P0 | 立即处理 + 通知王晓虎 |
| 核心功能异常 | P1 | 30分钟内 + 通知Jackson |
| 新需求 | 三条原则 | 评估后决定接/退/重设计 |
| 数据不一致 | 影响范围 | 判断P0/P1/P2 |
| 外部系统异常 | 依赖程度 | 检查连通性 + 告警 |
| 临时方案 | 是否变永久 | 标记截止时间 + 跟进 |
